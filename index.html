Dev Workflow - Rocket55

Manifesto

  • All code must be tracked in a repository
  • The entire wp-content folder should be tracked in this repository
  • Code on master branch MUST match code on production site
  • Unless building a brand new site, all code changes should be done in a branch off of master (git checkout -b yourbranchname)
  • All code changes to existing sites should use a Pull Request (or Merge Request if in GitLab) to add code to the master branch

Current development problems

  • Codebases in both GitLab and GitHub
  • No clear process for branching and merging in git
  • Numerous workflow issues, not the least of which being developers pushing code through an FTP client like Filezilla
  • Potential for overwriting other developers, and losing track of code (due to spotty use of Git, and using FTP software for pushing code to production)
  • No formal system for code reviews
  • Unorganized code & codebases (no consistent structure)
  • Different requirements and guidelines on nearly every project
  • No standard codebase to work from, due to the numerous clients and sometimes inheriting projects from other developers

Proposed solutions

  • Any new repository, and any code tracking, should be done in GitHub
  • The main branch for all repositories should be named main
  • Every single code change should take place within a pull request, no exceptions
  • Just kidding, there is one exception. If you are building a brand new site, you do not need to use pull requests or branches.
  • However, for any existing site, all code changes should be done in a PR. Yes, this adds a little bit of extra time on our end as developers, but it also safeguards against breaking changes, and allows us to revert a PR if bad code gets accidentally merged in or wasn't fully tested before merging
  • Not to be a nag, but we have to be strict with this. Every code change to an existing site needs to be in a PR. Once the code looks good, merge it into the staging branch. Once staging looks good, merge it into main

Future Roadmap

  • We should eventually strive to never be FTP-ing files onto a website. We should always set up some sort of continuous deployment (CI/CD), which allows our code changes to be pushed up directly from git and populate the site
  • We will need to research and further document solutions to this problem, but ideally we would use the tools provided by Codeship/WP engine/etc.
  • In addition, we should implement code reviews where necessary. Each PR will by default be tagged with a low risk flag, but if the code is medium or high risk, we recommend getting approval before merging. This may seem trivial now, but eventually as an agency, as we build more complex websites, and web apps over time, it will be essential to have this process built into our existing workflow.
  • It's very easy to get lazy and complacent with pushing code, and on marketing type of websites it is less imperative to have code reviews, but we should still get in the habit of keeping it as a viable option while building sites.

Rollout Plan

  • Get developer buy-in across the board
  • Effective (almost) immediately, all code should be tracked in Git, even for any new sites
  • Also, any new changes to code should be done in a pull request. Once that pull request is approved, either by its author, or by another developer, it should get merged back into the staging branch. Then once it's approved to go live, it should get merged into the main branch.
  • Test out CI/CD depoloyments on WordPress sites
